  ⏺ Both documents are ready. Here's a summary of NovelScript's key design decisions:

  Genuinely Novel Features

  - Bijections — Define a function and its inverse together; compose them with >>. No general-purpose language does this
  as a first-class construct.
  - Proof blocks — Inline tests as a language construct (not a framework). They run with your program, doubling as
  executable documentation.
  - Conditional pipeline |?> — Unwraps ok/some and short-circuits on err/none, eliminating nested error-handling
  boilerplate entirely.
  - Pattern matching as the only branching — No if/else/switch. One mechanism, always exhaustive, zero ambiguity about
  which construct to use.

  AI-Agent-Friendly Design Choices

  - Pipeline-first — Left-to-right data flow is easier to generate and verify than nested calls.
  - Contracts (require/ensure) — Agents can generate verifiable specs alongside code.
  - @intent annotations — Machine-readable purpose declarations.
  - Uniform syntax — Fewer special cases means fewer generation errors.
  - No null — Eliminates an entire bug class.

  Borrowed but Well-Integrated

  - Immutable-by-default with explicit mut and <- mutation
  - Structural typing via shapes
- Stream-based iteration (no loops)
  - Result/Option for error handling

  The spec is at docs/specification.md and the tutorial at docs/tutorial.md. Take a look and let me know what you'd like
  to change, add, or rethink before we start building the interpreter.
