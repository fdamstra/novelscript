  ---
  Specification Issues

  Truncated/Incomplete Content

  1. Section 1, Principle 8 is cut off (spec:28-29) — "No hidden conversions, no implicit returns from" ends
  mid-sentence.

  Missing from Type System

  2. Tuples are used but never defined — Tuple patterns appear in Section 6.3 ((a, b)), the Fibonacci example (line
  731-733), and the Stack.pop return type in the tutorial. But there's no Tuple entry in Section 3.2 (Composite Types).
  Need a formal definition with syntax for tuple types like (Num, Text).

  Operators/Syntax Not Documented

  3. // (floor division) missing from Section 2.5 — Introduced in Section 3.1 and present in the grammar (line 950), but
   not listed in the Operators and Punctuation table.
  4. // missing from precedence table (Section 2.6) — Should be grouped with *, /, % at precedence 6.
  5. >> (bijection composition) missing from Section 2.5 and 2.6 — Used in Section 9.3 but never listed as an operator.
  6. to and until missing from precedence table — They're keywords that act as infix operators for ranges, but their
  precedence relative to other operators is unspecified.

  Grammar Deficiencies (Section 15)

  7. Pipeline RHS too restrictive (line 944) — pipeline ::= conditional (('|>' | '|?>') IDENT call_args?)* only allows a
   bare identifier. It should allow dotted access like Circle.area and method chains like celsius_fahrenheit.rev.
  Something like IDENT ('.' IDENT)* call_args?.
  8. Trailing block lambda syntax missing from grammar — Section 4.5 (line 237) shows |> map do |n| ... end using
  |param| for lambda parameters, but the lambda production in the grammar (lines 961-962) doesn't describe this form at
  all.
  9. Map patterns missing from grammar — The pattern production (line 958) doesn't include map patterns, even though
  #{"key": v} patterns are shown in Section 6.3.
  10. with syntax unaccounted for — with(field: value) uses keyword-argument-like syntax that doesn't exist elsewhere in
   the language. The grammar doesn't describe it. Is with special syntax or a function? If it's a function, how do named
   field updates work?

  Standard Library Gaps

  11. unwrap, unwrap_or, unwrap_or_else — Used in tutorial (line 341-343) but missing from the spec's standard library.
  12. to_stream — Used in spec (line 688) but not in standard library.
  13. upper_first — Used in spec example (line 421) but not in standard library.
  14. each — Listed only as a stream terminal operation (line 711), but used throughout on lists directly. Should
  clarify that list operations implicitly work via streams, or add each to the List section.
  15. join — Listed only for streams (line 717) but used on lists in examples (line 425). Same issue as each.
  16. append on streams — Tutorial line 461 uses |> append(1) on a stream, but append is only defined for List[T] in
  Section 12.4.

  Semantic Ambiguities

  17. stream and collect are keywords AND function-like — Both are listed as keywords (line 56) but used with function
  call syntax: stream(0, n => n + 2), |> collect. The spec should clarify whether these are special forms that look like
   functions, or just built-in functions (in which case they shouldn't be keywords).
  18. Anonymous shape type syntax unexplained — Section 5.4 (line 337) shows type Point3D = { x: Num, y: Num, z: Num }
  using inline { field: Type } syntax. This anonymous record type isn't described anywhere else. Can it be used outside
  of type aliases?
  19. # in punctuation — Listed in Section 2.5 (line 81) but its role as the map literal prefix (#{...}) isn't explained
   in the operators section.

  ---
  Tutorial Issues

  20. Collatz example has a subtle problem (tutorial line 454-458) — Once the sequence hits 1, stream(n, x => ...) keeps
   producing 1 forever. The take_while(x => x != 1) excludes the final 1, then |> append(1) adds it back. This works but
   relies on append working on streams (not documented in stdlib). Also, stream here needs to produce values by
  iterating, but the stream would never naturally terminate — worth noting this is an infinite stream pattern.
  21. sort_by lambda with tuple destructuring (tutorial line 615) — sort_by((_, count) => -count) destructures a tuple
  in the lambda parameter. Tuple destructuring in lambda parameters isn't explicitly described in the spec.

  ---
  Cross-Document Consistency Issues

  22. return keyword behavior undocumented — Listed as a keyword (line 56), appears in the grammar (line 969), tutorial
  mentions it exists (line 91: "though it exists for early returns"), but the spec never describes its semantics. Can it
   be used in any do...end block? Only in functions? What about in lambdas?
  23. List vs Stream operation overlap — The spec puts map, filter, take, sort, etc. under Streams (Section 11.2), but
  examples throughout both documents use them directly on lists. The spec needs a clear statement about whether lists
  auto-convert to streams when piped into these operations, or whether these are overloaded for both types.

  ---
  Summary

  The most critical items to address:

  - Fix the truncated Principle 8 (easy)
  - Add Tuple to the type system (impacts type completeness)
  - Add // and >> to operator tables (they're used but invisible)
  - Fix the grammar — pipeline RHS, trailing block lambdas, map patterns, with syntax
  - Clarify List vs Stream — the biggest conceptual gap; it's unclear how they relate at runtime
  - Fill stdlib gaps — unwrap*, to_stream, upper_first, each/join on lists

  The language design itself is solid and coherent. These are mostly documentation gaps rather than design flaws.

  Specification Issues

  Grammar Gaps (Section 15)

  1. with_expr is unreachable — It's defined (line 1041) but never referenced from any other production. It should be
  reachable, likely as a pipe_target or primary alternative.
  2. option_pattern, result_pattern, shape_pattern are referenced but never defined — The pattern production (line
  1029-1031) uses these, but only list_pattern, map_pattern, and tuple_pattern have definitions. Need to add:
  option_pattern   ::= 'some' '(' pattern ')' | 'none'
  result_pattern   ::= 'ok' '(' pattern ')' | 'err' '(' pattern ')'
  shape_pattern    ::= IDENT '{' IDENT ':' pattern (',' IDENT ':' pattern)* '}'
  3. list, map, literal, call_args, annotation, comp_op, type all undefined — Referenced from productions but never
  defined. Acceptable for a "simplified" grammar, but call_args especially matters since its shape (does it include
  parens?) affects parsing. Worth defining at least the non-obvious ones.

  Void / Unit Value Ambiguity

  4. Void has "no literal" but () is used as a value — Section 3.1 says Void has no literal. But the early return
  example (line 246) uses case false => () as an expression producing the unit/void value. The tuple grammar (line 1044)
   requires at least 2 elements, and primary's '(' expression ')' requires an expression between parens — so () doesn't
  parse as either. () needs to be defined as the Void literal, or the example needs changing.

  Rest Pattern Without Binding

  5. [top, ..] in tutorial (line 404) vs grammar — The tutorial's peek function uses [top, ..] (rest pattern with no
  binding). But the grammar says list_pattern ::= '[' (pattern (',' pattern)* ('..' IDENT)?)? ']' — requiring an IDENT
  after ... Either the grammar should allow '..' alone (without a binding), or the tutorial should use [top, ..rest] or
  [top, .._].

  Principle 2 Wording

  6. "There is no else" vs else keyword — Principle 2 (line 15-16) says "There is no if, else, switch, or cond." But
  else IS a keyword used in match expressions. This is technically saying there's no standalone if/else statement, but
  the wording is confusing since else is actively used. Should say something like "no if/else statement" or "no if,
  switch, or cond".
